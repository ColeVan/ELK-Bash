# ELK Stack Deployment Scripts

## Overview
This set of scripts automates the deployment of the Elastic Stack (Elasticsearch, Logstash, Kibana) and Fleet Server on Debian-based Linux systems. It supports both single-node deployments and multi-node cluster setups for Elasticsearch.

## Script Structure

The deployment process is managed by an orchestrator script and several component-specific scripts:

-   **`common_functions.sh`**: A library of shared bash functions used by other scripts (e.g., for displaying messages, installing prerequisites). Not meant to be run directly.
-   **`elk_vars.conf`**: A configuration file automatically generated by the orchestrator (or the first script run standalone). It stores all user inputs, generated passwords, API keys, and certificate paths. **This file contains sensitive information and should be secured.**
-   **`deploy_elk_orchestrator.sh`**: The main script for deploying a new ELK stack. Use this for a fresh single-node setup or to deploy the first node of an Elasticsearch cluster.
-   **`install_elasticsearch.sh`**: Installs and configures the first Elasticsearch node. Generates SSL certificates (CA, node certs for all components) and relevant API keys. Can be run standalone but is primarily called by the orchestrator.
-   **`install_kibana.sh`**: Installs and configures Kibana, connecting it to the Elasticsearch instance. Called by the orchestrator.
-   **`install_logstash.sh`**: Installs and configures Logstash, setting up a default pipeline to receive Beats data and send it to Elasticsearch. Called by the orchestrator.
-   **`setup_fleet_server.sh`**: Installs the Elastic Agent on the ELK node and configures it as a Fleet Server. Also sets up example policies and Fleet outputs. Called by the orchestrator.
-   **`deploy_elasticsearch_node.sh`**: Used to add additional Elasticsearch nodes to an existing cluster. Requires an enrollment token generated by the first Elasticsearch node.

## System Prerequisites
-   A Debian-based Linux system (e.g., Ubuntu 20.04+).
-   `sudo` privileges for the user running the scripts.
-   Internet connectivity to download packages and Elastic Stack components.
-   Basic familiarity with shell scripts and the ELK Stack.
-   Required tools like `curl`, `tar`, `openssl`, `gpg`, `apt-transport-https`, `unzip`, `pv` (these are installed by `common_functions.sh` if missing).

## Deployment Scenarios

### Scenario 1: Deploying a Single ELK Node or the First Node of a Cluster
This is the recommended method for starting a new ELK deployment.

1.  **Clone/Copy Scripts**: Ensure all scripts (`deploy_elk_orchestrator.sh`, `install_*.sh`, `common_functions.sh`, `deploy_elasticsearch_node.sh`) are in the same directory on the target machine.
2.  **Make Scripts Executable**:
    ```bash
    chmod +x *.sh
    ```
3.  **Run the Orchestrator**:
    ```bash
    bash ./deploy_elk_orchestrator.sh
    ```
4.  **Follow Prompts**: The orchestrator will ask for necessary configuration details, such as:
    *   Deployment type (single/cluster).
    *   Elastic Stack version.
    *   IP addresses for components.
    *   Elasticsearch node name and superuser credentials.
    *   (If cluster) Total number of Elasticsearch nodes.
5.  **Completion**: The script will execute the component installations in order. Upon completion:
    *   `elk_vars.conf` will be populated in the current directory.
    *   Access Kibana at `https://${KIBANA_HOST}:5601` (Kibana host IP from your input).
    *   Superuser credentials are as you provided.
    *   If you chose a cluster deployment, enrollment tokens for additional Elasticsearch nodes will be saved in `enrollment_tokens.txt` and also within `elk_vars.conf`.

### Scenario 2: Adding Additional Elasticsearch Nodes to a Cluster
To add more Elasticsearch nodes after the first node has been set up (using Scenario 1 in "cluster" mode):

1.  **Obtain Enrollment Token**: From the first ELK node, get an enrollment token. This token was saved in `enrollment_tokens.txt` and `elk_vars.conf` during the first node's setup.
2.  **Prepare New Node**: On the new machine that will host the additional Elasticsearch node:
    *   Copy `deploy_elasticsearch_node.sh` and `common_functions.sh` to a directory.
    *   Make them executable: `chmod +x *.sh`.
    *   (Optional but Recommended) If you want the script to use the same Elastic version or superuser credentials (for fetching cluster info) as defaults, you can copy the `elk_vars.conf` file from your first ELK node to this directory on the new node. The script will still prompt for node-specific details like its IP and the enrollment token.
3.  **Run Node Addition Script**:
    ```bash
    bash ./deploy_elasticsearch_node.sh
    ```
4.  **Follow Prompts**: The script will ask for:
    *   The IP address for this new Elasticsearch node.
    *   A name for this new node.
    *   The Elastic Stack version (if not found in a copied `elk_vars.conf`).
    *   The IP address of an existing master-eligible node in your cluster.
    *   Elasticsearch superuser credentials (if not found in a copied `elk_vars.conf`).
    *   The enrollment token obtained in step 1.
5.  **Completion**: The script will install Elasticsearch, configure it, and join it to the cluster using the provided token. Monitor its logs and Kibana Stack Monitoring to confirm it has joined the cluster.

## Configuration (`elk_vars.conf`)
-   The `elk_vars.conf` file is crucial. It's created by `deploy_elk_orchestrator.sh` (or `install_elasticsearch.sh` if run standalone for the first time).
-   It stores:
    *   User-provided settings (IPs, version, primary username).
    *   Generated passwords for system users (e.g., `kibana_system`, `logstash_system`).
    *   Generated API keys (e.g., for Logstash output).
    *   Paths to generated SSL certificates (CA, Elasticsearch, Kibana, Logstash staging paths, etc.).
    *   Enrollment tokens (if applicable).
-   **Security**: Due to the sensitive information (passwords, keys), ensure `elk_vars.conf` is protected (e.g., `chmod 600 elk_vars.conf`) and not exposed.

## Running Individual Scripts (Advanced Users)
While the `deploy_elk_orchestrator.sh` is recommended for initial setup, individual component scripts (`install_elasticsearch.sh`, `install_kibana.sh`, etc.) can theoretically be run standalone.
-   If run standalone, `install_elasticsearch.sh` will prompt for all necessary initial inputs and create/append to `elk_vars.conf`.
-   Subsequent scripts (`install_kibana.sh`, etc.) depend on `elk_vars.conf` being populated with required variables from previous steps.
-   This approach is more complex and error-prone than using the orchestrator.

## SSL/TLS
-   The scripts automatically configure SSL/TLS for:
    *   Communication between Elasticsearch nodes (if clustered).
    *   Communication between Elasticsearch, Kibana, and Logstash.
    *   HTTPS access to Kibana.
    *   Secure input for Logstash (e.g., Beats input over SSL).
    *   Fleet Server communication with Elasticsearch and Elastic Agents.
-   A custom Certificate Authority (CA) is generated by `install_elasticsearch.sh`. The CA certificate and component certificates/keys are stored in locations defined in `elk_vars.conf` (typically under `/etc/elasticsearch/certs`, `/etc/kibana/certs`, `/etc/logstash/certs`, and a base staging directory like `/opt/elk_stack/generated_certs`).

## Troubleshooting
-   **Check `elk_vars.conf`**: Ensure it contains the expected values after the orchestrator or relevant script has run.
-   **Service Logs**:
    *   Elasticsearch: `sudo journalctl -u elasticsearch.service -f` or logs in `/var/log/elasticsearch/`.
    *   Kibana: `sudo journalctl -u kibana.service -f` or logs in `/var/log/kibana/` (as configured).
    *   Logstash: `sudo journalctl -u logstash.service -f` or logs in `/var/log/logstash/`.
    *   Elastic Agent (Fleet Server): `sudo journalctl -u elastic-agent.service -f` or logs in Elastic Agent specific paths (e.g., `/opt/Elastic/Agent/data/elastic-agent-*/logs/`).
-   **Configuration Files**: Review settings in:
    *   `/etc/elasticsearch/elasticsearch.yml`
    *   `/etc/kibana/kibana.yml`
    *   `/etc/logstash/logstash.yml`, `/etc/logstash/pipelines.yml`, pipeline configs in `/etc/logstash/conf.d/`
    *   Elastic Agent: Configuration is managed by Fleet, but local policy files might be in `/opt/Elastic/Agent/state/` or similar.
-   **Permissions**: Ensure correct file ownership and permissions, especially for certificate files and configuration directories. Scripts attempt to set these, but verification can be useful.
-   **Network**: Check firewall rules if components cannot communicate. Ensure specified ports (e.g., 9200, 9300, 5601, 5044, 8220) are accessible as needed between nodes and from your client machine.
-   **`set -e`**: The orchestrator uses `set -e`. If any command fails, the script will exit. Check the last few lines of output before the exit to identify the failing command.
